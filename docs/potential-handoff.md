## 기획 변경 요지

원 설계(§3.5)는 두 명령으로 쪼갰다 — `/잠재`(최다 재설정 단일 아이템 상세 비교) + `/잠재합계`(메소·큐브 횟수 랭킹).
**변경: 스타포스처럼 한 명령(`/잠재`) + 한 PNG 비교표**로 통합한다. 잠재 관련 지표를 캐릭터당 1행으로 모아 보여주는 게 깔끔하고,
[commands.py](../maple_mate/history/commands.py)의 표/부분성공/페이지네이션 자산을 그대로 재사용해 코드량도 준다. `/잠재합계`는 폐기.

## 실현가능성 검토 결과 (예시 이미지 3장 → 우리 API 대조)

실측 핵심([history.md](api/history.md) 라인 17·18·231): **cube·potential 둘 다 메소 소모량 필드가 없다.** 단 `item_level`·`item_equipment_part`·`cube_type`·`potential_type`·`item_upgrade_result`·`potential_option_grade`·`before/after_potential_option[].grade`·`upgrade_guarantee`는 있다.

| 지표 (출처) | 가능? | 방법 / 조건 |
|---|---|---|
| **총 큐브 횟수** (요구1, 예시2) | ✅ 즉시 | 기간 내 `cube_history` 건수. 추가 데이터 불필요 |
| **등업 횟수 + from-등급** (요구2) | ✅ (1건 검증) | `item_upgrade_result=="성공"` + before 등급. cube+potential 합산. "성공" 의미만 미니스파이크 확정 |
| **총 메소** (요구1, 예시1) | ⚠️ 데이터 게이트 | API에 금액 없음 → `item_level` 키로 **메소 재설정 단가표 룩업**. 단가표 미수령(G2) |
| 큐브종류별 횟수 (예시2) | ✅ 즉시 | `cube_type` 그룹 카운트. 단일 대상 보조 |
| 등급별 재설정 횟수, 잠재/에디 (예시3) | ✅ 즉시 | `potential_option_grade`·`additional_potential_option_grade` 그룹. 단일 대상 보조 |
| 누적 캐시·감정 비용 (예시1) | 🔴 스코프 밖 | 캐시 단가·감정비 테이블 필요, MVP 제외 |

요약: **"횟수" 계열은 전부 지금 구현 가능**, 막히는 건 **메소 금액 한 컬럼**뿐(스타포스가 비용공식을 받아야 했던 것과 동종의 데이터 게이트).

## 등업 메커니즘 (실측 근거 + 도출)

- **등업 판정** = `item_upgrade_result == "성공"`. 실샘플([B2_cube_yday.json](../spike/raw/B2_cube_yday.json))은 전부 **레전드리(최고등급) 아이템 → `"실패"`만** 나온다(최고등급은 등업 불가라 일관 — "성공=등급 상승" 해석과 모순 없음).
- **from-등급(어느 등급에서 올라갔나)** = 등업 직전 등급. `before_potential_option[0].grade`(또는 before 라인 최고 등급)가 모호함 없이 등업 전 상태를 준다. `potential_option_grade`는 성공행에서 before/after 어느 쪽인지 불명확 → **before 배열을 우선** 사용.
- 가능한 from-등급은 **3종뿐**: `레어→에픽` · `에픽→유니크` · `유니크→레전드리`(레전드리는 종착).
- 잠재 vs 에디 구분 가능(cube=`cube_type`/before·after 배열 비교, potential=`potential_type`). MVP는 **둘을 합산**해 from-등급 버킷에 넣고, 잠재/에디 분리는 단일 대상 보조에서만(D5).
- ⚠️ **미니스파이크 확정 1건**(G1): 우리 실데이터에 `"성공"` 레코드가 **0건**이라, ① "성공"=등급 상승 ② 천장(`upgrade_guarantee=true`) 등업 시 `item_upgrade_result` 문자열을 **실물 1건으로 확정**해야 한다. 등업 카운팅의 유일한 미검증 지점.

## 뱃지 렌더 결정 (D4 상세)

`/아이템`의 등급 뱃지 시스템이 이미 있다 — [item_card.py](../maple_mate/bot/item_card.py):
- `_GRADE`(라인 32): 레어=`(74,165,225)` 파랑 · 에픽=`(159,112,216)` 보라 · 유니크=`(240,190,52)` 금 · 레전드리=`(121,201,64)` 초록
- `_draw_pill`(라인 170): 색 테두리 + 옅은 채움 + 색 글자 **라운드 뱃지**

표 렌더러 [table_image.py](../maple_mate/bot/table_image.py)는 PIL 기반 + **셀 타입 시스템**(문자열·`Highlight`·`NumGrid`)이라, **pill 셀 타입(`GradeBadges`) 하나만 추가**(`_GRADE`·`_draw_pill` 공유)하면 표 칸에 등급 뱃지가 들어간다.

- **비교표 `등업` 컬럼**: from-등급별 색 뱃지 + 횟수, 0건 생략. 예: `[에픽 ×2][유니크 ×3]`(보라·금 pill). 색만으로 "어느 등급에서 올랐는지" 즉시 읽힘. 등업 0건이면 `—`.
- **단일 대상 보조**: `유니크 → 레전드리 ×3`처럼 FROM→TO 뱃지 진행(화살표).
- ⚠️ 우리 "뱃지"는 게임 내 등급 엠블럼 **스프라이트가 아니라** 등급명+색 라운드 pill(에셋 없음, 넥슨 스프라이트 스크랩 비권장). `/아이템`과 **동일 시각 언어** 유지.

## 스타포스와의 핵심 차이 (구현 시 주의)

- ✅ **레벨 매칭 불필요.** cube·potential 레코드에 `item_level`이 **직접 포함**(스타포스는 부재 → 3단 매칭이 필요했음). 메소 단가표만 있으면 **전건 직접 계산** → `M/N건` 미상 처리·`equipment_level.py`·`error_log(unmatched_equipment)` 불필요. 표에 `기준건수` 컬럼도 불필요.
- ✅ **두 엔드포인트 합산.** `/잠재`는 `history/cube` + `history/potential` 둘 다 페치(스타포스는 하나). 캐시 type 둘(`cube`·`potential_reset`).
- ⚠️ **메소 출처 = 메소 잠재 재설정(potential 엔드포인트)** 위주(§8). 큐브는 현금/이벤트라 메소 0 가정 가능(필요 시 메소 큐브 단가는 후속).

## 미해결 / 리스크

1. ⚠️ **"성공" 의미 미검증**(G1) — 등업 카운팅의 핵심 가정. 미니스파이크 전엔 구현하되 라이브로 1건 확인 후 확정.
2. ⚠️ **메소 단가표 미수령**(G2) — 메소 컬럼 보류. 단가표 형태(레벨 공식 vs 표)에 따라 `potential_cost.py` 모양 결정.
3. ⚠️ **닉 변경 시 `character_name` 불일치** — 과거 이력 누락 가능(스타포스와 동일 잔류).
4. potential 엔드포인트의 메소 재설정이 **등급 상승을 일으키는지** 불명(구버전 메소 재설정은 동급 내 재롤만 가능했음) — 등업은 cube가 주 출처일 수 있음. 미니스파이크에서 같이 관찰.
5. 큐브 메소(장인의/명장의) 비용 — MVP 0 가정. 친구 그룹이 메소 큐브를 쓰면 후속 단가 보강.
