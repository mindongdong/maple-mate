# ADR-0005 — 경험치 리더보드 누적 경험치 소스로 `ranking/overall` 단일 채택

- **상태:** 채택 (Accepted)
- **일자:** 2026-06-14
- **관련 문서:** [exp-leaderboard-work-order.md](../exp-leaderboard-work-order.md)(빌드 단위·G0 스파이크), [CONTEXT.md](../../CONTEXT.md)(조회 데이터 분류 — 종합 랭킹), [ADR-0001](0001-nexon-personal-key-model.md)(앱키 vs 개인키)
- **이력:** 경험치 리더보드 그릴링(2026-06-14) Q6 + 라이브 스파이크 G0([spike/verify_ranking.py](../../spike/verify_ranking.py), gitignore)로 확정. PR #16.

## 맥락 (Context)

경험치 리더보드는 등록 캐릭터들을 **누적 총 경험치**로 줄세우고, **어제 하루 획득량(Δ)**과 7일 추세를 보여야 한다. 넥슨 API에서 경험치를 얻는 길은 둘이다:

- **`character/basic`**(기존 `/스펙` 사용): `character_exp` = **현재 레벨 내** 경험치(예: 12,345,678) + `character_exp_rate`(%). **누적이 아니다** — 레벨이 다른 두 캐릭터를 직접 비교할 수 없고, 단일 비교 가능 숫자를 얻으려면 레벨→누적 EXP 환산 테이블(1~300)을 직접 소싱·유지해야 한다. 레벨업 구간 Δ 계산도 그 테이블 없이는 불가.
- **`ranking/overall?ocid=`**(미사용): `character_exp` = **누적 총 경험치**(예: 72,295,476,476,158) + 레벨 + **전체 서버 순위** + 길드/직업. 단일 비교 가능 숫자라 정렬이 곧 순위, Δ = `누적(D-1) − 누적(D-2)`로 레벨업이 자동 흡수된다.

둘 다 앱 키(공개·D-1)라 개인 키 없는 친구도 조회된다(ADR-0001 무관).

## 결정 (Decision)

**누적 경험치 소스를 `ranking/overall?ocid=&date=D-1` 단일로 채택한다.** 레벨→누적 EXP 테이블은 만들지 않는다.

1. 정렬 = `total_exp` 내림차순(히든 키). Δ = `누적(D-1) − 누적(D-2)`(이전 스냅샷 없으면 `—`, 음수는 클램프).
2. **종합 랭킹 미등재 캐릭터는 그날 제외**(`키 미등록`·`기록 없음`과 구별되는 부재 상태 — CONTEXT.md). EXP 테이블 폴백 없음.
3. **`exp%`만 예외적으로 `character/basic` best-effort 1콜**로 보강(`Lv.287 (45.2%)`). 이 콜의 실패·미준비는 행을 제외하지 않고 `Lv.287`로 그레이스풀 다운그레이드하며, 인클루전 게이트는 어디까지나 `ranking/overall` 성공 여부다.

### G0 스파이크로 확정한 사실 (실호출, 2026-06-14)

- `ranking/overall` `character_exp` = **누적**(72조@Lv287, D-8~D-1 단조비감소). 응답 `date`==요청 `date`(클램프 없음) → Δ 신뢰 가능.
- `ocid+date` 호출 시 응답 `ranking` 배열에 **대상 1건만**(`ranking[0]`이 곧 대상). 닉네임 매칭 불필요 → 닉 변경 매칭 리스크 소멸(CONTEXT.md 잔류 리스크 1건 해소).
- ⚠️ **`ranking/overall`은 `ocid`와 함께 `date` 필수** — date 무지정 시 HTTP 400(OPENAPI00004). `character/*`(date 무지정=최신)와 **반대**. 문서([api/ranking.md](../api/ranking.md))엔 선택으로 표기됨.
- D-1~D-8 전부 READY(12:47 KST 기준) → 8일 백필 가능.
- ⚠️ 필드명 함정: 랭킹 `character_guildname`(언더스코어 없음) vs basic `character_guild_name`.

## 검토한 대안 (Alternatives Considered)

- **`character/basic` + 레벨→누적 EXP 테이블(1~300)** — *기각.* 전 구간 EXP 테이블을 소싱·검증·유지해야 한다(스타포스 비용표 수준의 데이터 부채). `ranking/overall`이 누적을 직접 주므로 불필요. 전체 서버 순위도 못 얻는다.
- **`ranking/overall` + 미등재 시 basic/EXP테이블 폴백** — *기각.* 미등재(저렙·신규)는 친구 그룹에서 드물고(전원 고렙), 폴백을 위해 결국 EXP 테이블을 들여야 해 단일 소스의 이점이 사라진다. 미등재는 그날 제외로 충분.
- **누적 raw 숫자 표시** — *기각.* 14자리(수십조)는 가독성 0. `format_eok`도 억/만까지뿐. 대신 `ranking/overall`이 공짜로 주는 **전체 서버 순위(#)**를 표시.

## 결과 (Consequences)

**긍정**
- 레벨→누적 EXP 테이블이라는 데이터 부채를 **회피**. 누적·순위·Δ가 한 콜에서 파생.
- `ranking[0]` 직접 사용으로 닉 매칭 불필요 → CONTEXT.md 잔류 리스크("닉 변경 시 이력 `character_name` 불일치")가 이 기능에선 무효.
- 개인 키 무관(앱 키) → 등록자 전원 포함이 자연스러움.

**부담 / 잔류**
- **종합 랭킹 미등재 캐릭터는 리더보드에서 그날 빠진다** — 저렙·신규·비활성. 친구 전원 고렙이라 실관측 드묾. 진짜 미등재 응답형태(빈 배열 가정)는 첫 저렙 등록자 등장 시 라이브 확정 필요.
- **D-1 readiness 시점 의존** — 잡은 10:00 KST지만 G0는 12:47 확인. 10:00에 D-1 미준비면 그 캐릭만 그날 제외(상수 1곳으로 시각 조정 가능).
- **Δ는 종합 랭킹 exp 갱신 타이밍에 의존** — 비활동일 Δ=0(정상), 갱신 배치 지연이 관측되면 보조 지표(최근 7일 합) 백로그.
- **`exp%`는 2번째 소스(`character/basic`) 의존** — Q6 "단일 소스" 정신과 약한 상충이나, best-effort라 실패해도 핵심(순위·Δ) 무손상. EXP 테이블을 들이지 않으므로 ADR의 핵심 결정은 유지.
